# Как оформить листинги кода в отчёте

Гайд по тому, как вставить исходники C++ в отчёт по проекту-калькулятору.
Покрывает три варианта: **LaTeX (Overleaf)**, **Word**, **PDF из исходного `.md`**.

---

## 1. Что вставлять и в каком объёме

Не нужно тащить **весь** код проекта в отчёт — это 1015 строк, никто не
будет читать. Оптимум — **ключевые куски с пояснением**, всего 100–200
строк листингов.

### Рекомендуемый набор для варианта 2

| № | Файл | Что показать | Зачем |
|---|------|--------------|-------|
| 1 | `src/main.cpp` | целиком (~30 строк) | точка входа, фиксация локали |
| 2 | `src/parser.cpp` | `parse_expression` (~20 строк) | приоритет `+ −` |
| 3 | `src/parser.cpp` | `parse_term` (~22 строк) | приоритет `× ÷`, деление на 0 |
| 4 | `src/parser.cpp` | `parse_factor` (~30 строк) | скобки, унарный минус, диспетчер |
| 5 | `src/parser.cpp` | `apply_function` (~20 строк) | четыре функции варианта 2 |
| 6 | `src/parser.cpp` | `parse_number` (~30 строк) | локаль-независимый разбор числа |
| 7 | `src/ui/main_window.cpp` | `compute()` (~15 строк) | главный сценарий + try/catch |
| 8 | `src/ui/main_window.cpp` | `keyPressEvent()` (~15 строк) | поддержка клавиатуры |
| 9 | `src/ui/format.cpp` | `format_result()` (~15 строк) | вывод дробной части |

Под каждым листингом — **1–2 предложения**, что эта функция делает и
зачем она нужна. Не нужно повторять то, что и так видно из кода.

### Что **не** показывать

- `theme.cpp` (QSS-стили) — никому не интересно.
- `button_layout.cpp` (массив кнопок) — упомянуть в тексте, листинг не
  нужен.
- Заголовочные файлы (`.h`) с тривиальными объявлениями — достаточно
  показать `.cpp`.
- `parser_tests.cpp` — упомянуть как факт «есть 41 юнит-тест», листинг
  не нужен.

## 2. Вариант LaTeX / Overleaf (рекомендую)

В Overleaf листинги делаются пакетом `listings` — он умеет подсветку
синтаксиса C++, нумерацию строк, рамки, переносы длинных строк.

### Преамбула — добавить в начало `main.tex`

> `literate` не нужен.

### Способ 1 — листинг прямо в текст

```latex
\begin{lstlisting}[language=C++, caption={Главный сценарий вычисления выражения}, label={lst:compute}]
void MainWindow::compute() {
    QString expr = expression_edit_->text().trimmed();
    if (expr.isEmpty()) {
        show_result("", false);
        return;
    }
    try {
        double r = calc::evaluate(expr.toStdString());
        history_label_->setText(expr + " =");
        show_result(format::format_result(r), false);
    } catch (const std::exception& e) {
        history_label_->setText(expr);
        show_result(QString::fromStdString(e.what()), true);
    }
}
\end{lstlisting}

### Способ 2 — подключение целого файла

```latex
\lstinputlisting[
    language=C++,
    caption={src/main.cpp — точка входа программы},
    label={lst:main},
    firstline=1, lastline=58            % опционально: диапазон строк
]{../src/main.cpp}
```

Удобно: при правке исходника листинг автоматически обновится при
следующей компиляции.

### Способ 3 — приложение «Полный листинг»

Если препод хочет видеть **весь** код — оформи это как **Приложение А**
в конце отчёта (после Заключения). Один `\lstinputlisting` на каждый
исходный файл, без сокращений.

```latex
\appendix
\section{Полный исходный код программы}

\subsection{src/main.cpp}
\lstinputlisting[language=C++]{../src/main.cpp}

\subsection{src/parser.h}
\lstinputlisting[language=C++]{../src/parser.h}

\subsection{src/parser.cpp}
\lstinputlisting[language=C++]{../src/parser.cpp}

% ... и так далее по всем файлам
```

## 3. Вариант Microsoft Word

Word не умеет подсветку C++ из коробки, поэтому есть три пути.

### Путь 1 — моноширинный шрифт + рамка (быстро, без подсветки)

1. Скопируй код из исходника (из VSCode / любого редактора).
2. Вставь в Word (Ctrl+V).
3. Выдели вставленный блок.
4. Установи шрифт **Consolas** или **Courier New**, размер 9–10 pt.
5. Меню **Главная → Границы → Внешние границы** (рамка вокруг блока).
6. Меню **Главная → Заливка → Светло-серый** (для фона).

Минус: нет цветовой подсветки. Но для отчёта-самоделки сойдёт.

### Путь 2 — копировать с подсветкой из VS Code (рекомендую) ⭐

В VS Code есть встроенный «Copy with Syntax Highlighting»:

1. Открой исходник в VS Code.
2. Выдели нужный фрагмент.
3. Открой Command Palette (`Cmd+Shift+P` / `Ctrl+Shift+P`).
4. Введи и запусти **«Copy With Syntax Highlighting»**.
5. Вставь в Word — цвета сохранятся, шрифт тоже.
6. Добавь рамку вручную (как в пути 1).

### Путь 3 — экспорт через `pandoc` (для тех, кто умеет в командную строку)

Если уже пишешь отчёт в Markdown:

```bash
pandoc report.md -o report.docx --highlight-style=tango
```

`--highlight-style=tango` (или `pygments`, `kate`, `breezedark`) даёт
цветную подсветку C++ в `.docx`-файле.

### Подпись к листингу в Word

После каждого блока кода сделай **отдельную строку** с подписью:

```
Листинг 1 — Главный сценарий вычисления выражения (compute)
```

И **до** листинга вставь предложение, что показывает этот код:

> Главный обработчик кнопки «=» приведён в листинге 1. Метод читает
> строку выражения, обрезает пробелы, передаёт парсеру и в зависимости
> от исхода (успех или исключение) выводит результат зелёным или текст
> ошибки — красным.

## 4. Где какие листинги показать в отчёте

Согласно task.md, в отчёте есть разделы «Проектирование», «Разработка»,
«Тестирование». Привязка листингов:

### В разделе «Проектирование»
- Листинг 1: `src/main.cpp` целиком — показать структуру
  Qt-приложения: `QApplication`, фиксация локали, создание `MainWindow`.
- Листинг 2: фрагмент `button_layout.cpp` (5–6 строк из массива) —
  показать, как раскладка хранится как данные.

### В разделе «Разработка» — самое главное
- Листинг 3: `parse_expression` — приоритет `+ −`, реализация цикла
  накопления результата.
- Листинг 4: `parse_term` — приоритет `× ÷`, проверка деления на 0.
- Листинг 5: `parse_factor` — диспетчер по типу токена, унарный минус,
  скобки.
- Листинг 6: `apply_function` — четыре функции варианта 2 с проверкой
  области определения.
- Листинг 7: `parse_number` — локаль-независимый разбор числа.
- Листинг 8: `MainWindow::compute()` — связка UI и парсера.

Перед каждым листингом — ссылка на соответствующую блок-схему.
Например: «Логика функции `parse_expression` приведена в листинге 3,
её блок-схема — на рисунке 5».

### В разделе «Тестирование»
- **Листинги не нужны.** Вместо них — таблица «вход → ожидаемый
  результат → получено» и скриншоты работающего калькулятора.
- Можно упомянуть: «корректность парсера проверяется 41 юнит-тестом
  в `tests/parser_tests.cpp`, все тесты проходят».

### В Приложении А (если нужно)
- **Полный код всех файлов** (`main.cpp`, `parser.h`, `parser.cpp`,
  `ui/*`). Каждый файл — отдельным подразделом, через
  `\lstinputlisting`.

## 5. Оформление подписей по ГОСТ

В вузовских отчётах подписи к листингам обычно делают так:

```
Листинг 3 — Функция parse_expression (приоритет операторов + и −)
```

- Слово **«Листинг»** с заглавной, **дефис** (—) после номера.
- **После подписи** — не точка, а просто конец строки.
- В тексте отчёта на листинг ссылаются как «листинг 3», «(см. листинг 3)».
- Шрифт подписи — обычный, размер как у основного текста.

Аналогично для рисунков (включая блок-схемы): «Рисунок 5 — Блок-схема
функции parse_expression».

## 6. Чек-лист перед сдачей

- [ ] Все листинги пронумерованы по порядку (Листинг 1, 2, 3...).
- [ ] У каждого листинга есть подпись с описанием.
- [ ] Перед каждым листингом — фраза в тексте, что он показывает.
- [ ] Шрифт листинга моноширинный (Consolas / Courier / DejaVu Sans Mono).
- [ ] Размер шрифта листинга меньше основного (обычно 9–10 pt против 12–14).
- [ ] Кириллица в комментариях C++ читается (особенно проверить в Word —
      бывают проблемы кодировок при копировании).
- [ ] В оглавлении отражены приложения, если есть «Полный код».
- [ ] Длинные строки кода либо переносятся (`breaklines=true`), либо
      сами не вылезают за поля (проверь визуально).

## 7. Минимальный пример куска отчёта (LaTeX)

```latex
\subsection{Реализация парсера}

Парсер построен по принципу \emph{рекурсивного спуска}. Для каждого
уровня приоритета операторов написана отдельная функция, которая
вызывает функцию следующего (более сильного) уровня. Это естественно
выражает грамматику выражения через грамматику языка C++.

Самый верхний уровень — функция \texttt{parse\_expression}, обрабатывающая
сложение и вычитание (листинг~\ref{lst:expr}). Её блок-схема приведена
на рисунке~\ref{fig:flow_expr}.

\begin{lstlisting}[language=C++,
                   caption={Функция parse\_expression — приоритет + и -},
                   label={lst:expr}]
double parse_expression() {
    double left = parse_term();
    while (true) {
        char c = peek();
        if (c != '+' && c != '-') break;
        eat();
        char nxt = peek();
        if (nxt == '+' || nxt == '*' || nxt == '/') {
            throw std::runtime_error(
                std::string("Двойной оператор: '") + c + nxt + "'");
        }
        double right = parse_term();
        left = (c == '+') ? left + right : left - right;
    }
    return left;
}
\end{lstlisting}

Функция читает первый операнд через \texttt{parse\_term} (умножение/деление),
а затем в цикле подбирает все следующие пары «оператор + операнд».
Проверка \texttt{nxt} защищает от ввода вида \texttt{2++3}: после
бинарного оператора ожидается операнд, а не ещё один оператор.
```

## TL;DR

1. **Не суй весь код в основной текст** — только ключевые функции с
   пояснением. Полный код — в Приложение А.
2. **LaTeX/Overleaf** — `\lstinputlisting` или окружение `lstlisting`.
3. **Word** — копируй из VS Code через «Copy With Syntax Highlighting».
4. Под каждым листингом — подпись «Листинг N — описание».
5. В тексте — ссылка на листинг (и на блок-схему той же функции).
